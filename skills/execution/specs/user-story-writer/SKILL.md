# User Story Writer Skill

## Overview

This skill generates well-structured user stories with acceptance criteria and edge cases from feature descriptions, PRDs, BRDs, or raw notes. It produces output in either .docx or .md format based on user preference.

## Workflow

### Step 1: Understand the Input

The user may provide context in several forms:

- **Feature description or brief** — a paragraph or bullet points describing what needs to be built
- **PRD or BRD** — a structured document (possibly generated by another skill). If the user uploads a .docx file, read it using the docx skill approach.
- **Raw/verbal notes** — unstructured text, meeting notes, or rough ideas

When the user provides input, assess what you have and what's missing.

**Must-have context (ask if missing):**
- What is the feature or product being described?
- Who are the users/personas involved? (You need at least one clear user type to write "As a [user]" stories)
- What are the core things the user should be able to do?

**Nice-to-have context (ask if missing, but can proceed without):**
- Business context or goals behind the feature
- Known constraints (technical, platform, regulatory)
- Existing user flows or system behaviors this interacts with
- Priority or phasing information

**How to ask:**
- Don't dump all questions at once. Read what's been shared and only ask for what's clearly missing.
- If the user provides a PRD or BRD, extract the answers from the document first. Only ask about gaps.
- If the user says "just work with what I've given you," proceed and mark assumptions clearly.

### Step 2: Configuration Choices

Before generating, ask the user two things using the `ask_user_input_v0` tool:

**Question 1 — Acceptance criteria format:**
- Given-When-Then (BDD style)
- Simple checklist (bullet points)

**Question 2 — Story organization:**
- Grouped under epics/features (hierarchical)
- Flat list (no grouping)

**Question 3 — Output format:**
- Word document (.docx)
- Markdown file (.md)

Present all three as a single batch of questions.

### Step 3: Generate User Stories

#### Story Format

Every user story must follow this structure:

```
**Story ID:** [PREFIX-001]
**Story:** As a [specific user/persona], I want [specific goal/action], so that [clear benefit/reason].

**Acceptance Criteria:**
[Format based on user's choice — see below]

**Edge Cases / Negative Scenarios:**
- [What happens when things go wrong or inputs are unexpected]
```

#### Story ID Convention
- Use a short prefix derived from the feature name (e.g., "CST" for Claim Status Tracker, "OB" for Onboarding)
- Number sequentially: CST-001, CST-002, etc.

#### Acceptance Criteria Formats

**If user chose Given-When-Then:**
```
- Given [precondition/context],
  When [action/trigger],
  Then [expected outcome].
```

**If user chose Simple Checklist:**
```
- [ ] [Specific, testable condition that must be true]
- [ ] [Another condition]
```

#### Edge Cases / Negative Scenarios

For each story, include 1-3 relevant edge cases. These should cover:
- Invalid or unexpected inputs
- Error states and failure modes
- Boundary conditions
- Permission/access edge cases
- Empty states or no-data scenarios

Not every story needs extensive edge cases. Use judgment — a simple display story might have one edge case, while a form submission story might have several.

#### Grouping (if hierarchical)

When the user chose grouped organization:

```
## Epic: [Epic Name]
[One-line description of the epic]

### Feature: [Feature Name]
[One-line description]

[Stories under this feature]
```

When the user chose flat list, just output stories sequentially with no grouping headers.

### Content Rules

- **NEVER fabricate user personas, flows, or requirements.** Only generate stories from what the user has provided.
- **Be specific.** "As a user, I want to use the feature" is useless. "As a policyholder, I want to check my claim status via WhatsApp by sending my policy number" is useful.
- **One action per story.** Don't combine multiple behaviors into a single story.
- **Acceptance criteria must be testable.** A developer or QA person should be able to read each criterion and definitively say whether it passes or fails.
- **Mark assumptions.** If you're inferring something not explicitly stated by the user, prefix it with `[Assumption]` so the user can validate.
- **Include a summary count** at the end: total stories, total edge cases, and any areas where you felt context was insufficient.

### Step 4: Generate the Output File

**If .docx:** Read `/mnt/skills/public/docx/SKILL.md` for docx creation best practices. Follow the same formatting principles as the PRD generator skill:
- US Letter page size, 1-inch margins
- Arial font, 12pt body
- Heading 1 for epics (if grouped), Heading 2 for features/story IDs
- Proper bullet lists using LevelFormat.BULLET
- Clean, minimal formatting
- Title page with: "User Stories — [Feature/Product Name]", Date, Version 1.0

**If .md:** Generate a clean markdown file with proper heading hierarchy, code-block-free acceptance criteria, and standard markdown lists.

**Filename:** `User_Stories_<FeatureName>_v1.docx` or `User_Stories_<FeatureName>_v1.md`

Save to `/mnt/user-data/outputs/` and use `present_files` to share.

### Step 5: Summary

After generating, provide:

```
📋 Generated X user stories across Y epics/features
🧪 Included Z edge cases / negative scenarios

Stories are based on the context you provided. Here's what I'd flag:

⚠️ Assumptions made:
- [List any assumptions you marked in the stories]

❓ Areas that could use more context:
- [Specific gaps — e.g., "No info on error handling for payment failures"]
```

Keep the summary concise. The document speaks for itself.

## Important Notes

- If the input is a PRD or BRD from another skill, extract context intelligently. Don't just regurgitate sections — translate requirements into actionable stories.
- If the user asks to "add more stories" or "cover this additional flow," generate only the new stories and offer to append them to the existing document or create a new one.
- Stories should be independent enough that a developer could pick up any single story and understand what to build without reading all the others.
- Don't pad the output with generic stories. If the input describes 5 distinct user actions, write 5-8 stories (with edge cases), not 25 stories full of filler.
